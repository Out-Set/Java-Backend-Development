Lets observe that whether only 1-thread (event loop) handles multiple requests at the same time with blocking i/o call.
=======================================================================================================================
Agenda: Everything on consumer-app
----------------------------------
1. Restrict event loop (thread) creation to 1 only, setting below in vm argument.
	-Dreactor.netty.ioWorkerCount=1 
	(in intellij: Run -> Edit Configurations -> Select your application -> Modify options -> Add VM opitons)
2. Run application in debug mode to observe the thread & variables.
3. Hit the blocking api: http://localhost:8081/consume which further calls: http://localhost:8080/hello (blocking in nature: gives response after 10 seconds sleep).
4. Call other api: http://localhost:8081/consumer, 3-4 times frequently.
5. Go and check the logs, you will see only 1-thread (i.e. or-http-epoll-1) is handling all the requests in non-blocking (no-waiting - same thread is executing the all api calls simultaneously) nature.
6. Go and check the thread & variables, you will see only 1-thread (i.e. reactor-http-epoll-1) is created.
7. Hence it is proved that the single thread continues accepting, dispatching, and resuming multiple request pipelines simultaneously. This behavior is possible because WebFlux uses reactive programming + non-blocking I/O (NIO).

Conclusion:
===========
Spring WebFlux
--------------
 -> Single event-loop thread can process many requests concurrently
 -> No blocking at thread level (unless you explicitly introduce blocking)
 -> Threads are shared across reactive pipelines
 -> Scaling is based on event-loop efficiency, not thread count

Spring MVC (Spring Web):
========================
 -> Uses Servlet + Tomcat thread-per-request model
 -> Each request occupies a thread until completion
 -> Causes thread starvation when calls block (database, REST calls, file I/O)
 -> Needs a large thread pool to handle many concurrent requests

Summary:
--------
 -> Setting Reactor Netty to a single thread and still observing multiple concurrent requests being processed proves the non-blocking nature of Spring WebFlux.
 -> This would not be possible in Spring MVC, where a single blocking request would hold up a thread and prevent it from serving others.

import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;
import org.springframework.web.reactive.function.client.WebClient;
import reactor.core.publisher.Mono;

import java.time.Duration;

@RestController
public class ProducerController {

    WebClient webClient = WebClient.builder().baseUrl("http://localhost:8080").build();

    @GetMapping("/consume")
    public Mono<String> consumeData() {
        System.out.println("executing consumeData() method... ");
        // rest call here of producer app
        Mono<String> response = webClient.get().uri("hello").retrieve().bodyToMono(String.class);
        return response;
    }

    @GetMapping("/consumer")
    public Mono<String> consumer() {
        System.out.println("executing consumer() method... ");
        return Mono.just("consumer data... ").delayElement(Duration.ofSeconds(2));
    }

    @GetMapping("/simple")
    public Mono<String> simpleCall() {
        System.out.println("executing simpleCall() method... ");
        return Mono.just("simple data... ");
    }
}
